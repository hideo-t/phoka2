# PR #1 画像マッピング誤り 事後分析

**日付**: 2026-05-13
**影響範囲**: phoka2 / `cases/index.html` (PR #1 commit `84356e7`)
**症状**: 12 事例カードのうち多くで「タイトル」と「メイン画像」の組み合わせが無関係。

## 根本原因

`C:\Users\hidex\parkhomes_data\site_data.json` のスキーマには **per-section grouping が存在しない**。
スクレイパーがページ DOM を平坦化しており、`pages[i].images[]` は配列の順序が大体 DOM 順
ではあるものの、「どの画像がどの `<h2>` セクションに属するか」を直接示すフィールドがない。

各画像エントリは次のキーを持つ:
```
{
  "original_src": "...",
  "local_file": "<hash>.jpg",
  "size_bytes": ...,
  "alt": "",          # ほとんど空
  "section_heading": "",  # ほとんど空 (致命的)
  "context_text": "",     # ほとんど空 (時々前後のテキストが入る程度)
  ...
}
```

致命的なのは `section_heading` が常に空白だったこと。スクレイパーが DOM 構造から
セクションヘディングを抜き取って各画像に紐付ける処理を実装していなかった。

## PR #1 の試み (失敗の原因)

`scripts/extract_cases.py` の `pick_images()` は次のような位置ベースのヒューリスティクス:

```python
per = max(1, len(mains) // total_titles)  # ≒ 252 / 46 = 5
start = position * per
sliced = mains[start:start + max_n]
```

**なぜ動かなかったか:**

1. **per-section の画像枚数が均等ではない**:
   元サイトは事例によって画像 1 枚〜10 枚と大きく差がある。
   - `senaga-glamping`: 1 枚
   - `plus-garden-cafe`: 10 枚
   - `crystal-villa-ingya`: 9 枚
   平均 5 枚で区切ると、最初のいくつかの事例は遅延し、後ろの事例は前にずれる。
2. **位置の起点が不明**:
   `case-1` ページの先頭画像 (idx=0, idx=1) はページバナー + サムネ一覧であり、
   実画像とは別カテゴリ。スキップ判定を `mode=crop` と `size < 30KB` で行ったが、
   一部の正規メイン画像も小さい (50KB 程度) ものがあり、フィルタが粗かった。
3. **画像配列の順序が DOM 順と一致しない可能性**:
   Jimdo ページの一部は画像レイジーロードや遅延注入で、スクレイパーが取得した
   `<img src=...>` 列が画面表示順と異なる場合がある。スクレイパー側で明示的な
   visual-order ソートをしていない。

つまり「タイトルリストの順序」と「画像配列の順序」が緩く対応していたものの、
**毎事例について実際にどの画像が割り当たっていたかは、データだけから復元不可能**だった。

## 影響範囲

PR #1 で配置された 12 事例 × 3 画像 = 36 画像のうち、**おそらく過半数が誤割当**。
（visual review 未実施で push してしまった）。

## 対策 (PR #2 で実施)

### 短期対策
- `parkhomes_data/site_data.json` は信頼できないものとして扱い、画像マッピングには
  使用しない。
- 元サイト DOM を手元で再パースして取得した **正解マッピング**
  (`_import/review/cases_image_truth.json`) を真実の起点とする。
- 画像は jimcdn から直接再取得 (User-Agent はオーナー再取得を明記、Crawl-Delay 5s)。

### 中長期対策 (再スクレイプ時の改修事項)
スクレイパーを次のスキーマに対応させる:

```json
{
  "url": "...",
  "title": "...",
  "sections": [
    {
      "heading": "【PLUS GARDEN CAFE】",
      "heading_dom_index": 7,
      "paras": ["..."],
      "images": [
        {"local_file": "...", "alt": "...", "original_src": "..."}
      ]
    }
  ]
}
```

DOM 取得時に `<h1>`/`<h2>`/`<h3>` を境界として画像を区切ること。Jimdo の場合は
具体的には `.j-clear-line` ブロック等のグルーピング境界を見るとよい。

### 検証手順の追加
今後は **「画像と見出しが目視で合っているか」を 12 件全部スクリーンショットで確認**
してからマージする運用に変更すること。本件は PR #1 の "目視確認" を省略したのが
最終的なミスのトリガー。

## 教訓

- スクレイパーの「全部 flat な配列」出力は、後段でグルーピングを再構築できる保証がない。
  上流で構造を保ったまま吐くか、DOM 取得時に section 境界を必ず付与すること。
- LLM の summary 生成は title だけでも見られる出力を出すため、画像が壊れていても
  気付きにくい。**画像と本文の意味的整合性は別チェックレイヤーが必要**。
